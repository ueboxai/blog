UE 项目版本控制实战：Git + 虚幻盒子的完整工作流

Unreal Engine 项目使用 Git 版本控制面临独特挑战：二进制资产（.uasset、.umap）无法合并、仓库体积庞大、LFS 配置复杂。本文基于 3 年大型 UE 项目经验，提供一套完整的 Git + 虚幻盒子解决方案。

一、UE 项目 Git 配置基础

1.1 为什么 UE 项目需要特殊配置

与传统代码项目不同，UE 项目包含大量二进制资产：

  • .uasset - 材质、纹理、蓝图等资产文件
  • .umap - 关卡文件
  • .uproject - 项目配置文件
  • 导入的原始资源（FBX、PSD、WAV 等）

这些文件无法像代码一样 diff 和 merge，需要 Git LFS（Large File Storage）管理。

1.2 初始化配置清单

创建 .gitattributes 文件：

# UE 资产文件使用 LFS
*.uasset filter=lfs diff=lfs merge=lfs -text
*.umap filter=lfs diff=lfs merge=lfs -text
*.uproject filter=lfs diff=lfs merge=lfs -text
*.udk filter=lfs diff=lfs merge=lfs -text

# 原始资源文件
*.fbx filter=lfs diff=lfs merge=lfs -text
*.obj filter=lfs diff=lfs merge=lfs -text
*.psd filter=lfs diff=lfs merge=lfs -text
*.png filter=lfs diff=lfs merge=lfs -text
*.jpg filter=lfs diff=lfs merge=lfs -text
*.wav filter=lfs diff=lfs merge=lfs -text
*.mp3 filter=lfs diff=lfs merge=lfs -text
*.mp4 filter=lfs diff=lfs merge=lfs -text

# 构建产物（不应该提交）
Binaries/ export-ignore
Build/ export-ignore
DerivedDataCache/ export-ignore
Intermediate/ export-ignore
Saved/ export-ignore
*.sln export-ignore

二、Git LFS 深度配置

2.1 LFS 存储配额管理

GitHub LFS 免费配额 1GB，大型项目很快会用完。解决方案：

  • 使用 Git LFS + 自建存储（MinIO、AWS S3）
  • 或使用虚幻盒子的资产托管服务

配置自建 LFS 服务器：

# 配置 LFS 服务器
 git config lfs.url "https://lfs.yourcompany.com"

# 或者在 .lfsconfig 文件中
[lfs]
    url = "https://lfs.yourcompany.com"

2.2 LFS 文件迁移

已有项目迁移到 LFS：

# 1. 安装 git-lfs
 git lfs install

# 2. 追踪现有文件
 git lfs track "*.uasset"
 git lfs track "*.umap"

# 3. 迁移历史提交（重写历史）
 git lfs migrate import --include="*.uasset,*.umap"

# 4. 强制推送（谨慎操作！）
 git push --force

三、分支策略与协作工作流

3.1 推荐的分支模型

基于 Git Flow adapted for UE：

  • main - 稳定版本，随时可发布
  • develop - 开发分支，日常集成
  • feature/* - 功能分支，如 feature/new-weapon
  • release/* - 发布分支，如 release/v1.2
  • hotfix/* - 紧急修复分支

3.2 UE 特有的分支注意事项

关卡编辑冲突是最大痛点。解决方案：

  1. 关卡分区：将大关卡拆分为多个子关卡，不同人负责不同区域
  2. 数据层分离：Gameplay 数据放在 Data Table，不直接存在关卡中
  3. 锁定机制：编辑关卡前在团队频道声明，避免同时编辑
虚幻盒子的资产锁定功能可以自动检测谁正在编辑哪个关卡，并在尝试编辑时给出警告。

四、二进制资产冲突解决

4.1 冲突不可避免

当两个人同时修改同一个 .uasset 或 .umap 文件时，Git 无法自动合并，会产生冲突。

4.2 冲突解决流程

# 1. 拉取最新代码
 git pull origin develop

# 2. 发现冲突
# Auto-merging Content/Weapons/Rifle.uasset
# CONFLICT (content): Merge conflict in Content/Weapons/Rifle.uasset

# 3. 查看冲突文件
 git status

# 4. 使用 UE 编辑器解决（见下文）
# 5. 标记为已解决
 git add Content/Weapons/Rifle.uasset

# 6. 完成合并
 git commit -m "Resolve conflict in Rifle.uasset"

4.3 UE 编辑器中的冲突解决

UE 提供了资产合并工具（需启用 Source Control 插件）：

  1. 打开编辑器，连接 Git 源码控制
  2. 在 Content Browser 中找到冲突文件
  3. 右键 → Source Control → Merge
  4. 使用可视化工具选择保留哪个版本

如果没有启用 Source Control 插件，可以手动：

  1. 备份冲突文件
  2. 从 Git 历史中提取两个版本的文件
  3. 在 UE 中分别打开，手动对比合并

六、CI/CD 集成

6.1 自动化构建

配置 GitHub Actions 自动构建：

name: UE Build

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: windows-latest
    steps:
    - uses: actions/checkout@v3
      with:
        lfs: true

    - name: Setup UE
      uses: unreal-actions/setup-unreal@v1
      with:
        version: '5.3'

    - name: Build Project
      run: |
        "C:\Program Files\Epic Games\UE_5.3\Engine\Build\BatchFiles\Build.bat" ^
          MyProjectEditor Win64 Development ^
          "%~dp0MyProject.uproject" -waitmutex

总结

UE 项目的 Git 版本控制需要特别注意：

  1. 正确配置 Git LFS 管理二进制资产
  2. 建立合理的分支策略，避免关卡冲突
  3. 掌握二进制资产冲突的解决方法
  4. 使用虚幻盒子等工具增强版本控制能力
